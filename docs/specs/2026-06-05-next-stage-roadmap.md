# NL2DSL 下一阶段路线图：语义可靠性与治理闭环

> 日期：2026-06-05
> 阶段定位：从“功能可跑通”推进到“语义可量化、链路可治理、试点可落地”。

---

## 1. 背景判断

NL2DSL 当前已经具备较完整的工程底座：DSL First、Governance First、LangGraph 查询管道、权限注入、审计日志、RAG、Semantic Optimizer、Evaluation Framework 等模块均已形成。

但从复杂自然语言查询分析可以看到，系统目前的主要瓶颈不是“有没有查询链路”，而是复杂业务语义的稳定理解能力：

- 复合过滤容易丢失，例如“华东线上金额大于 5000”。
- 数值范围、否定条件、时间语义支持不足，例如“5000 到 20000 之间”“非手机品类”“1 月份”。
- 隐式 JOIN 推导不稳定，例如“按供应商”“按客户名称”“按城市等级”。
- 高级分析语义缺口明显，例如“各品类销售额最高的产品”“各品类销售额占总销售额比例”。
- 置信度、优化器报告、审计追踪还需要进一步产品化，避免只在测试中可见。

因此下一阶段不建议优先扩展更多演示功能，而应聚焦“语义可靠性与治理闭环”。

---

## 2. 阶段目标

下一阶段的核心目标：

> 让每一次自然语言查询都能被校验、被优化、被解释、被评估、被追责，并显著提升复杂业务问法的语义正确率。

具体目标：

| 目标 | 说明 |
|------|------|
| 语义质量可量化 | 将复杂查询失败样本沉淀为评测集，按 Semantic / Planning / Execution / Governance 四类指标持续跟踪 |
| 优化器进入主链路 | Semantic Optimizer 不只停留在单测中，而是在真实查询 trace 中输出 Fix / Warn / Reject |
| 复杂查询能力提升 | 优先补齐数值范围、否定、时间、隐式 JOIN、分组 TopN、占比等高价值语义 |
| 治理证据可见 | 每次查询都能说明权限过滤、敏感字段处理、SQL 扫描、沙箱预检等治理动作 |
| 生产试点可准备 | 支持单领域试点演示，具备稳定部署、基线评估、问题回溯能力 |

---

## 3. 六周路线

### 第 1 周：质量债清理与基线建立

目标：先确保现有链路稳定，不在已知缺陷上继续叠功能。

重点任务：

- 复核并关闭 `docs/reports/code_review_report.md` 中的 P0 / P1 问题。
- 修复配置相对路径问题，避免 `configs/intents.yaml` 依赖当前工作目录。
- 修复多对象 compare 聚合只比较前两个对象的问题，明确支持 N 路对比或显式限制为双对象对比。
- 检查 API 是否正确透传 `warning`、`clarification`、`error` 等状态，避免部分成功被包装成完全成功。
- 清理用户可见解释中的内部字段，例如 `__sub_query_id`。
- 跑通当前全量测试与 evaluation baseline，保存基线结果。

验收标准：

- 全量单元测试、集成测试、E2E 测试通过。
- evaluation baseline 可稳定生成。
- 代码审查报告中的 P0 问题均有修复或明确结论。
- 从非项目根目录启动服务时，意图配置仍可正常加载。

---

### 第 2 周：复杂过滤语义攻坚

目标：优先解决复杂查询中最常见、收益最高的过滤条件丢失问题。

重点任务：

- 支持数值比较语义：
  - “金额大于 5000”
  - “金额小于 3000”
  - “单笔金额超过 8000”
- 支持范围语义：
  - “5000 到 20000 元之间”
  - “近 7 天到近 30 天”
- 支持否定语义：
  - “非手机品类”
  - “不是线上渠道”
- 支持多条件组合：
  - “华东地区 + 线上渠道 + VIP 客户 + 金额大于 8000”
- 将复杂查询报告中的相关失败案例转成 evaluation dataset。

验收标准：

- `Filter` 维度评估分数明显提升。
- 复杂过滤类用例不再丢失关键条件。
- DSL 中过滤字段、操作符和值均来自语义层或 value_map，不引入自由文本字段。

---

### 第 3 周：时间语义与隐式 JOIN

目标：补齐业务查询中高频的时间表达和跨表维度解析能力。

重点任务：

- 支持基础时间表达：
  - “1 月份”
  - “本月”
  - “上月”
  - “最近 7 天”
  - “最近 30 天”
- 支持对比时间表达：
  - “同比”
  - “环比”
  - “去年同期”
- 完善时间缺失澄清：
  - “销售额趋势”
  - “销售额增长了多少”
- 增强隐式 JOIN 推导：
  - “按供应商统计销售额”
  - “按客户名称统计消费金额”
  - “按城市等级统计销售额”
- 将 JOIN 推导结果写入 optimizer report 或 trace。

验收标准：

- `Time`、`Join`、`Planning` 相关评估分数提升。
- 缺少时间上下文的趋势/增长类查询进入澄清，不静默猜测。
- 需要跨表维度的查询能自动注入 JOIN，无法唯一推导时返回可解释的 Warn 或 Clarification。

---

### 第 4 周：高级分析语义

目标：支持更接近真实 BI 分析的查询模式。

重点任务：

- 支持分组 TopN：
  - “各品类销售额最高的产品”
  - “每个地区订单量前 3 的品牌”
- 支持占比语义：
  - “各品类销售额占总销售额的比例”
  - “线上渠道各品牌销售占比”
- 明确 DSL 扩展方式：
  - 是否引入 `window` / `rank` / `derived_metrics` / `post_process` 等结构。
  - 禁止绕过 DSL 直接从自然语言生成 SQL。
- 评估 SQLBuilder、PostProcessor、Planner 三者的职责边界。
- 为高级分析查询补充 docs 与 evaluation case。

验收标准：

- 分组 TopN 不再退化为全局排序。
- 占比查询返回比例字段，并能解释分母口径。
- 新增 DSL 结构有文档、有校验、有测试，不破坏治理链路。

---

### 第 5 周：评估、反馈与可观测闭环

目标：让系统具备持续改进能力，而不是靠人工感觉判断“准不准”。

重点任务：

- 打通 optimizer on/off 的评估对比。
- 打通 fallback / LLM on/off 的评估对比。
- 按 domain 输出独立评估结果，例如 ecommerce、bank、supply_chain。
- 将复杂查询失败样本、用户反馈、审计 trace 关联起来。
- 设计反馈处理流程：
  - 高频纠错模式提取。
  - corrected_dsl 转为评测用例。
  - 反馈样本进入 RAG few-shot 或语义层配置更新。
- 输出一份固定格式的质量报告。

验收标准：

- 一条命令能生成当前版本评估报告。
- 能看到 optimizer 开启前后的收益对比。
- 每个修复都能对应至少一个新增或更新的 evaluation case。
- 反馈数据能追溯到原始查询、生成 DSL、最终 SQL 和执行结果。

---

### 第 6 周：生产试点准备

目标：选择一个领域完成可演示、可治理、可回溯的试点版本。

重点任务：

- 选择一个优先试点领域，建议先从 ecommerce 开始。
- 固化试点领域的语义层配置、权限配置、测试数据和评测集。
- 补齐部署文档：
  - 环境变量。
  - 数据库连接。
  - Milvus Lite / Milvus Server 切换。
  - LLM Mock / Real 模式切换。
- 补齐运行手册：
  - 如何启动。
  - 如何跑评估。
  - 如何查看审计。
  - 如何定位一次查询失败。
- 准备试点演示脚本：
  - 简单汇总查询。
  - 复杂过滤查询。
  - 时间趋势查询。
  - 跨表 JOIN 查询。
  - 权限受限查询。
  - 澄清问题查询。

验收标准：

- 单领域试点可稳定跑通。
- 查询链路具备完整 trace。
- 治理动作可解释。
- 评估报告可复现。
- 失败查询有明确排查路径。

---

## 4. 优先级排序

| 优先级 | 方向 | 理由 |
|--------|------|------|
| P0 | 质量债清理 | 已知 P0 不关闭会污染后续评估与试点 |
| P1 | 复杂过滤、时间、JOIN | 直接对应当前复杂查询失败主因 |
| P1 | Evaluation baseline | 没有基线就无法证明修复有效 |
| P2 | 高级分析语义 | 对产品体验重要，但应在基础语义稳定后推进 |
| P2 | 治理可观测 | 生产试点必需，但可与语义修复并行补齐 |
| P3 | 前端体验优化 | 当前优先级低于语义质量和治理闭环 |

---

## 5. 不建议优先做的事

下一阶段暂不建议优先投入以下方向：

- 大规模重做前端页面。
- 新增大量演示型查询模板。
- 在没有 DSL 扩展设计的情况下直接支持复杂 SQL。
- 绕过语义层硬编码指标、维度或权限。
- 只优化 Prompt，不补评测和规则。
- 只追求 E2E 全绿，不分析语义正确率。

---

## 6. 阶段完成定义

当满足以下条件时，可以认为“语义可靠性与治理闭环阶段”完成：

- 复杂查询评测集覆盖当前已知失败类型。
- 复杂查询语义正确率较当前报告显著提升，目标至少达到 15/22。
- Semantic Optimizer 在真实查询链路中稳定输出优化报告。
- Evaluation Framework 能输出可复现的版本质量报告。
- 权限、脱敏、SQL 扫描、查询沙箱、审计日志在 trace 中可见。
- 至少一个业务领域具备生产试点条件。
