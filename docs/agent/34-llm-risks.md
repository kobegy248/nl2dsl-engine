# 34. LLM 层风险与缓解（企业落地）

## 34.1 成本

**问题：** 每次查询都调 LLM，高并发时 API 费用可观。

**缓解方案：**
- 相似查询 DSL 缓存（问题文本嵌入相似度 > 0.95 直接复用）
- 小模型做意图分类，仅复杂查询调大模型
- 设置用户/租户级日调用配额

## 34.2 延迟

**问题：** LLM 响应 1-3s，加后续链路总延迟 3-5s。

**缓解方案：**
- 异步生成 + 流式返回（先返回 DSL，再返回 SQL，最后返回数据）
- 预生成热门查询 DSL（定时任务分析审计日志，缓存 Top-100 查询）
- LangGraph 节点级超时（RAG 1s / LLM 生成 5s / 自检 3s）

## 34.3 幻觉

**问题：** LLM 编造不存在的字段或指标。

**缓解方案：**
- RAG 注入准确的语义层上下文（只提供已注册字段）
- DSL 校验器强校验（字段白名单、指标存在性）
- LangGraph 自检节点（LLM 二次检查字段合理性）
- 修正重试机制（最多 3 次，每次带错误反馈）

## 34.4 稳定性

**问题：** LLM 服务偶发超时或错误。

**缓解方案：**
- 多模型降级策略（主模型失败自动切备用模型）
- 断路器模式（连续失败 5 次后暂停调用，30s 后恢复）
- 异步重试队列（非关键查询失败后可后台重试）

## 34.5 版本漂移

**问题：** 换模型后 DSL 生成质量变化，原有 Prompt 可能失效。

**缓解方案：**
- 回归测试集（固定 50-100 个问题，验证 DSL 正确率）
- Prompt 版本管理（与模型版本绑定）
- A/B 测试切换（新模型并行运行，对比准确率后再全量切换）
