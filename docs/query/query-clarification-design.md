# Query Clarification 设计

## 设计目标

在用户查询存在歧义时，不猜测、不编造，主动返回澄清问题让用户确认。

核心原则：**低置信度路由到澄清，而非静默执行低质量查询。**

## 检测条件

ClarificationDetector 分析用户输入，检测以下 4 类歧义：

### 1. 时间缺失

**触发条件**：问题包含时间敏感关键词（趋势/变化/增长/下降/同比/环比/最近/走势/波动），但**没有**明确时间范围。

**示例**：
- ❌ "销售额趋势" → 触发澄清（缺少时间范围）
- ✅ "本月销售额趋势" → 不触发（有明确时间）

**返回选项**：`["本月", "上月", "最近7天", "最近30天", "全部"]`

### 2. 指标歧义

**触发条件**：问题中包含语义层未精确定义的多义词。

| 歧义词 | 可选含义 |
|--------|---------|
| 销量 | 支付订单量 / 发货数量 / 完成数量 |
| 客户数 | 注册用户 / 下单用户 / 支付用户 |

**示例**：
- ❌ "各品类销量" → 触发澄清（"销量"含义不明）
- ✅ "各品类支付订单量" → 不触发（语义层已注册）

### 3. 维度歧义

**触发条件**：问题中包含位置/时间类多义词。

| 歧义词 | 可选含义 |
|--------|---------|
| 地区 | 收货地址 / 发货仓库 / 注册地 |
| 时间 | 下单时间 / 发货时间 / 支付时间 |

### 4. 比较基准歧义

**触发条件**：问题涉及增长/下降/同比/环比，但**没有**明确对比基准。

**示例**：
- ❌ "销售额增长了多少" → 触发澄清（环比？同比？）
- ✅ "销售额同比增长" → 不触发（基准明确）

**返回选项**：`["环比上月", "同比去年", "不对比，只看当前值"]`

## 与 LangGraph 集成

ClarificationDetector 是 LangGraph 管道的**第一个节点**（`clarification`）：

```
START
  │
  ▼
clarification（歧义检测）──[需要澄清]──> END（返回澄清响应）
  │[无需澄清]
  ▼
plan（意图分类）
```

当 `detect()` 返回非空列表时，查询立即终止，API 返回：

```json
{
  "status": "clarification",
  "message": "查询存在歧义，请确认以下信息",
  "items": [
    {
      "type": "time_missing",
      "question": "请确认时间范围",
      "options": ["本月", "上月", "最近7天", "最近30天", "全部"]
    }
  ]
}
```

用户通过 `POST /api/v1/query/resume` 提交澄清答案后，查询继续执行。

## 扩展方式

新增歧义类型无需改代码结构，只需在对应字典中添加条目：

```python
# nl2dsl/query/clarification.py
_METRIC_AMBIGUOUS: dict[str, list[str]] = {
    "销量": ["支付订单量", "发货数量", "完成数量"],
    # 新增：
    "客单价": ["平均订单金额", "平均支付金额"],
}
```

## 与 PostProcessor 的关系

Clarification 发生在**查询执行前**（语义理解阶段）。
PostProcessor 发生在**查询执行后**（结果处理阶段），处理 "TOP-1 per group" 等 SQL 难以表达的语义。

两者互补：Clarification 解决"问得不清楚"，PostProcessor 解决"结果需要加工"。
