# 13. 业务规则与同义词处理

## 13.1 同义词歧义

**问题：** 不同用户对同一业务术语理解不同。例如"收入"可能指销售额、营收、毛利或净利润；"上周"可能指自然周或业务周。

**影响：** LLM 无法确定用户意图，生成错误的 DSL，导致查询结果不符合用户预期。

**缓解方案：**

### 13.1.1 术语表映射

在 `configs/terms.yaml` 中维护标准术语与别名映射：

```yaml
terms:
  sales_amount:
    aliases: ["销售额", "收入", "销售金额"]
    metric: sales_amount
    description: "含税销售额，不含退款"
```

### 13.1.2 Prompt 显式注入

将术语表注入 System Prompt，让 LLM 优先使用标准术语。

### 13.1.3 歧义时反问

当置信度低时，返回多个可能的解释供用户选择：

```json
{
  "status": "ambiguous",
  "options": [
    {"term": "sales_amount", "description": "含税销售额"},
    {"term": "net_revenue", "description": "税后净营收"}
  ]
}
```

### 13.1.4 反馈闭环学习

记录用户选择，自动优化术语映射。

## 13.2 语义层风险与缓解（企业落地）

### 13.2.1 指标口径战争

不同部门对"销售额"定义不同（含税？退款？）。

**缓解：** YAML 配置 + 版本控制 + 审批流程；指标血缘追踪。

### 13.2.2 配置维护成本

表结构变更后 YAML 没同步，导致查询失败。

**缓解：** 自动化检测（对比数据库元数据 vs YAML）；CI 校验。

### 13.2.3 复杂指标不支持

同比环比、留存率、漏斗等无法用简单聚合表达。

**缓解：** 扩展 DSL 支持 window function、CTE、子查询。
